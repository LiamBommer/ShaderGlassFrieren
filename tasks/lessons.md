# Lessons

- 当用户描述“底图”和“晕影”的透明关系时，要先区分被显示的 ImageTexture 层与叠加在上方的 ChromaFlow 层；opacity 应绑定到用户指定的合成层，而不是仅修改控件文案。
- 对透明度类参数必须验证两个边界：opacity=100 时上层完全遮住底图，opacity=0 时底图完全可见。
- 控制面板的章节顺序应跟随用户的操作优先级，上传入口需要放在标题下的第一个章节。
- 当底层组件没有直接暴露用户要求的参数时，应使用最接近的可用属性实现可解释的近似，并在控件说明中明确“近似”而不是把它伪装成原生能力。
- 如果只能在项目层增加视觉衰减包络，必须区分“原生 field 密度衰减”和“额外可见度衰减”，并把默认值设为不改变原效果，避免误称为修改了底层模拟时间。
- 使用包含 `VideoTexture` 的 Shader 栈时，不能只依赖 `opacity=0` 来屏蔽空视频：最终 pass 仍可能读取未就绪的外部纹理并跳过整张画布。无媒体时应将节点设为 `visible: false`，上传视频后再启用，并在浏览器中验证无媒体状态下 Swirl/ChromaFlow/FlutedGlass 仍可见。
## 2026-07-17

- 有声媒体的 `autoplay` 受浏览器策略控制：初始化时应保持实际音量和未静音状态，并在首次普通用户交互时重试播放；不能把“播放失败”误认为“已静音”。
- 初始静音需求要同时同步状态变量、隐藏 checkbox、可见按钮的 `aria-pressed` 和视觉 class；音量滑块的用户事件需要显式取消静音，预设/重置的程序化更新则不能误触发取消静音。
- 调整媒体纳入范围时，除了新增 `.gitignore` 规则，还要从未推送的历史中移除被排除文件；`.gitattributes` 只保留实际需要使用 LFS 的媒体，避免远程仓库继续包含不需要的素材。
- 修改文件后必须先检查 `git status` 和 `git diff`，确认改动已经 commit，再判断是否已经 push 到远程；工作区里的 `M` 不代表远程仓库已经更新。
